# 项目维护总纲

## 1. 文档定位

本文件只保留当前仍然长期有效的维护约束，不再混入阶段性迁移记录。

- Phase 1 历史记录已归档到 `docs/archive/maintenance-requirements-phase1.md`
- Phase 2 实施记录归档到 `docs/archive/requirements-phase2.md`
- 当前若进入新阶段，应新增对应阶段文档，例如 `docs/requirements-phase3.md`
- 若阶段目标发生变化，优先更新对应阶段文档；若长期约束变化，再更新本文件

## 2. 当前产品定位

本项目当前定位为：

- 本地优先的跨平台桌面音乐播放器
- 由自建 Spring Boot 4 后端提供数据、配置与文件访问能力
- 由 JavaFX 桌面端承载播放器 UI 与交互
- 客户端只调用自建后端，不直接依赖第三方音乐平台接口
- 默认运行形态应保持轻量，不要求使用者额外安装独立数据库服务

## 3. 长期约束

### 3.1 合规约束

- 不再以网易云或其他未授权第三方音乐平台作为正式产品基础
- 不再新增任何高风险抓取、直链解析、未授权下载逻辑
- 后端只处理用户自有、导入、扫描到或明确获得授权的内容
- 任何“兼容旧接口结构”的设计，只能用于迁移期降低改造成本，不能改变上述边界

### 3.2 技术基线

- JDK 17+
- Maven 多模块
- 后端：Spring Boot 4
- 桌面端：JavaFX
- 默认数据库：SQLite
- 必须保留 MySQL / PostgreSQL 扩展能力

补充说明：

- 默认开发与开源运行形态按 `Spring Boot 4 + SQLite` 设计
- SQLite 的目标是保留“开箱即用”的轻量体验，而不是把项目默认前提改成必须先安装数据库

### 3.3 架构边界

- `player-server` 负责 REST API、配置、数据库、文件访问、存储适配与业务编排
- `player-fx` 负责 UI、播放控制、窗口交互与用户操作反馈
- `player-model` 负责桌面端与服务端共享的轻量业务模型
- `player-common` 负责通用响应、JSON、路径与基础工具

严格禁止：

- JavaFX 直接访问数据库
- UI 层直接拼 SQL、文件路径或第三方网络请求
- 将历史 legacy 适配实现继续扩散进新核心结构

当前阶段说明：

- 本地播放、本地歌词、本地导入等桌面链路已基本迁完，不是第二阶段的主要改造面
- 第二阶段主要改造面是把原先“在线模块”的桌面交互链路统一收进 `player-server`
- 为减少 UI 改动，后端可以继续提供兼容旧客户端预期的接口结构
- `player-fx` 本地音乐目录与 `player-server` 在线模块存储目录必须分离，不能混用同一套文件目录语义

### 3.4 跨平台与存储

- 必须支持 Windows 与 macOS
- 路径与文件操作必须使用 `Path` / `Paths` / `Files`
- 默认本地存储目录、数据库路径、缓存路径必须可配置
- 不允许把仅 Windows 可用的命令或路径写法变成主流程前提

当前默认约定：

- `player-fx` 本地音乐继续使用相对路径 `./LocalMusic/Music`、`./LocalMusic/Lrc`
- `player-server` 在线模块默认使用相对路径 `./runtime/online/music`、`./runtime/online/lyrics`、`./runtime/online/covers`
- 在线模块元数据默认进入 `./runtime/musicbox.db`

### 3.5 UI 与体验底线

- 新版 UI 不得退化成普通 JavaFX 默认控件堆砌
- 必须持续保留 Material Design 倾向与旧版核心交互语义
- 主窗口、底部播放条、详情抽屉、桌面歌词、迷你模式、托盘能力都属于正式产品范围
- 后续重构可以优化实现，但不能破坏现有核心交互

### 3.6 代码质量

- 优先控制大类和高耦合逻辑继续膨胀
- 新增依赖必须优先选择仍在维护、许可证清晰、无明显高危问题的库
- 能自研且可控的 UI 与适配层能力，优先自研
- 每轮重要改动后至少完成编译验证；涉及启动链路时要做实际启动验证

## 4. 当前维护规则

- 长期结论写进本文件
- 阶段性目标、迁移策略、接口清单写进对应阶段文档
- 已过期的阶段过程记录不继续堆在主文档中，应归档处理
