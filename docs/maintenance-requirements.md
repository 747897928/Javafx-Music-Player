# 项目维护需求文档

## 1. 当前结论

本项目后续维护方向已经明确：

- 完全移除网易云相关抓取、搜索、歌词、下载、播放直链等能力
- 不再依赖任何存在明显版权或服务条款风险的第三方非官方音乐接口
- 项目重构为多模块架构
- 后端使用 Spring Boot 4
- 桌面端使用 JavaFX
- 技术基线采用 JDK 17+

这意味着原项目的 `cloudutils` 相关逻辑只可作为历史代码参考，不能作为后续产品基础。

## 2. 产品定位

目标产品不再是“依赖外部音乐平台的在线播放器”，而是：

- 本地优先的跨平台桌面音乐播放器
- 可由本地后端提供统一数据和配置能力
- 可扩展但不绑定任何特定在线音乐来源

默认运行方式：

- 本地启动 Spring Boot 4 后端
- 本地启动 JavaFX 桌面客户端
- 客户端只调用自建后端
- 前后端分开启动

服务范围：

- 必须支持本机桌面客户端访问
- 必须支持局域网访问
- 允许后续扩展到广域网访问

## 3. 核心约束

### 3.1 法律与合规

- 必须彻底移除网易云相关侵权风险逻辑
- 禁止继续保留“默认可用”的第三方未授权抓取模块
- 即使未来扩展在线源，也必须通过合法授权或明确合规的数据来源接入
- 后端仅处理以下来源的内容：
  - 用户自己导入的内容
  - 用户自己扫描到的本地内容
  - 用户自己上传的内容
  - 使用者拥有或明确获得授权的内容
- 不提供未授权的音乐文件和歌词内容

### 3.2 技术基线

- JDK：17 起步
- 后端：Spring Boot 4
- 前端：JavaFX，依赖通过 Maven 管理
- 构建工具：Maven 多模块
- JavaFX 启动入口应采用独立 launcher 包装，避免直接以继承 `Application` 的主类作为普通入口导致运行时组件缺失问题

### 3.3 平台支持

- 必须支持 Windows
- 必须支持 macOS
- 路径处理不能写死 Windows 风格路径
- 所有文件路径必须使用 `Path` / `Paths` / `Files`

### 3.4 数据库支持

- 必须支持 SQLite
- 必须支持 MySQL
- 必须支持 PostgreSQL
- 默认数据库为 SQLite

### 3.5 本地默认存储

- SQLite 默认文件必须使用工程相对路径
- 本地音乐、歌词、封面、缓存等目录必须可配置
- 默认配置应保证 Windows 和 macOS 均可运行

## 4. 架构目标

建议采用以下多模块结构：

- `player-parent`
  - 聚合父工程
- `player-common`
  - 常量、通用异常、工具类、基础注解、统一返回模型
- `player-domain`
  - 实体、值对象、仓储接口、领域服务接口
- `player-infra`
  - 数据库访问、文件系统访问、配置实现、外部适配
- `player-server`
  - Spring Boot 4 应用
  - REST API
  - 配置管理
  - 数据持久化
- `player-fx`
  - JavaFX 桌面客户端
  - 只调用后端 API，不直接操作数据库

可选扩展模块：

- `player-provider-local`
  - 本地音乐扫描与导入适配
- `player-provider-remote`
  - 合规远程源扩展点，仅保留接口，不默认实现高风险来源

## 5. 前后端职责边界

### 5.1 后端职责

- 管理曲库元数据
- 管理音乐文件访问与分发
- 管理歌单、收藏、历史记录、播放队列快照
- 管理歌词、封面、缓存索引
- 统一文件路径与应用配置
- 提供数据库兼容层
- 对客户端提供 REST API

### 5.2 桌面端职责

- 播放器 UI
- 播放控制
- 桌面歌词
- 迷你模式
- 托盘能力
- 调用后端 API 获取数据与配置

### 5.3 严格禁止

- JavaFX 直接访问数据库
- JavaFX 继续直接耦合旧 `CloudRequest` / `CloudMusicSpider`
- 在 UI 层直接拼接本地路径、SQL、网络请求

## 6. 功能需求

### 6.1 第一阶段必须保留

- 本地音乐扫描
- 本地音乐播放
- 歌词显示与同步
- 专辑封面读取
- 基础歌单管理
- 最近播放
- 搜索本地曲库
- 迷你模式
- 系统托盘

### 6.2 第一阶段建议保留

- 专辑封面写回音频文件
- 桌面歌词
- 播放模式切换
- 本地导入与批量导入

### 6.3 第一阶段必须移除

- 网易云搜索
- 网易云歌单发现
- 网易云歌词抓取
- 网易云歌曲下载
- 网易云歌曲播放直链解析

## 7. 非功能需求

### 7.1 代码质量

- 代码必须满足 Sonar 扫描要求
- 禁止超大类继续膨胀
- 单个类职责必须单一
- 方法长度和复杂度应受控
- 需要明确异常边界

### 7.2 可读性

- 代码风格要统一
- 类名、方法名、字段名要表达业务含义
- 复杂逻辑必须写简洁注释
- 公共接口、核心服务、重要流程必须写文档注释
- 适当使用 Spring / Lombok / Jakarta 等注解提升可读性，但不能滥用

### 7.3 UI 质量基准

- 新版 UI 不能弱于 `README.md` 中展示的旧版效果
- 必须保留明显的 Material Design 倾向
- 必须保留清晰的层次感、阴影感、卡片感和交互动效
- 主窗口、详情抽屉、歌词页、精简模式、关于弹窗都必须达到“开源项目可展示”的完成度
- 不能退化成普通 JavaFX 默认控件堆砌页面
- 如果改动原有视觉风格，最终效果必须明确优于旧版
- 底部当前歌曲信息区必须保留“点击后展开详情抽屉页”的核心交互
- 详情抽屉页必须具备平滑的上滑展开/收起动画，不能退化成普通弹窗或普通切页
- 底部歌词按钮必须恢复为“桌面歌词面板”的开关，不能继续误用为抽屉开关
- 桌面歌词必须是独立的半透明悬浮面板，且保持置顶与可拖拽
- 桌面歌词默认弹出位置应锚定主播放器窗口，优先出现在播放器下方，而不是固定到屏幕底部
- 迷你模式必须按旧版 `SimplifyModelStage` 的语义重建，而不是简单缩小主窗口
- 迷你模式必须独立于主窗口显示，不能因为主窗口隐藏而跟着一起消失
- 主窗口初始尺寸不得写死为大屏常量，必须根据当前屏幕可视区域计算
- 主界面、抽屉页、歌单卡片区至少应具备第一版响应式缩放能力，避免在较小屏幕上直接溢出
- 推荐歌单区域在较宽窗口下应支持展示 8 张或更多卡片，不能长期固定只显示 6 张
- 迁移阶段说明卡片不应继续出现在正式界面中
- 抽屉的作用范围应为主窗口中间内容区，不应把顶部标题栏和底部固定播放条一起作为抽屉内容
- 抽屉关闭态必须被完整裁剪，不能在底部、侧边或标题栏附近泄露任何隐藏内容

### 7.4 可维护性

- 所有模块必须有清晰边界
- 领域模型不能依赖 UI
- UI 层不能侵入基础设施层实现细节
- 后续每次维护前必须先阅读本需求文档
- 编码开始前必须先核对本需求文档是否与最新结论一致
- 每次新增重要设计决策后，必须同步更新本需求文档
- 若实现过程中发现原有结论不成立，必须先修正文档再继续实现

### 7.5 跨平台

- 文件操作必须兼容 Windows 与 macOS
- 不允许依赖仅 Windows 可用的命令行为主流程
- 托盘、打开目录、定位文件等平台差异能力必须封装

## 8. 依赖策略

### 8.1 JavaFX

JDK 11 之后 JavaFX 已从 JDK 中拆出，因此本项目必须通过 Maven 显式引入 OpenJFX 依赖。

结论：

- 可以通过 Maven 正常管理 JavaFX
- 这是当前标准做法

### 8.2 JFoenix

当前判断：

- 原始 JFoenix 项目长期未活跃
- 即使存在第三方维护版本，也不能直接默认其为稳定长期基础

维护策略：

- 允许调研 `us.ihmc:jfoenix`
- 但在没有完成兼容性与可维护性验证前，不把它视为默认技术选型
- UI 风格优先保持 Material Design
- 如果替代方案无法达到不弱于当前项目的视觉质量，则不应轻易放弃现有 Material 风格路线

建议优先级：

1. 优先评估是否用现代 JavaFX 原生组件加自定义样式实现 Material 风格
2. 评估 `MaterialFX` 这类更现代的 JavaFX Material 组件库
3. 最后再评估 `us.ihmc:jfoenix`

原因：

- UI 风格可以自己控制
- 被长期无人维护的 UI 库反向绑定风险很高
- 如果存在更现代且活跃的 Material 风格库，应优先考虑

### 8.3 当前项目对 JFoenix 的实际依赖

经代码盘点，当前项目实际用到的 JFoenix 相关能力主要是：

- `JFXButton`
- `JFXSlider`
- `JFXTabPane`
- `JFXListView` / `JFXListCell`
- `JFXPopup`
- `JFXDrawer`
- `JFXDecorator`
- `JFXDepthManager`
- `SVGGlyph`

结论：

- 当前项目不是“全站深度绑定 JFoenix 全家桶”
- 但确实依赖其少数关键组件带来的 Material 风格和交互体验
- 真正难替代的不是普通按钮和滑块，而是 `JFXDrawer`、窗口装饰、图标系统和部分交互动效

### 8.4 当前代码中的关键使用点

- `JFXDrawer`：底部抽屉式歌词/详情页
- `JFXPopup`：音量弹出层、精简模式弹出层
- `JFXTextField`：带浮动标签的搜索输入
- `JFXTabPane`：主内容区页签容器
- `JFXDecorator`：关于窗口、确认窗口、主窗口顶栏装饰
- `SVGGlyph`：大量图标绘制
- `ToasterFX`：导入、下载、复制等非阻塞冒泡提示

### 8.5 替代方案评估结论

当前推荐方案不是继续长期绑定 JFoenix，也不是退回到“纯原生 JavaFX + 很普通的样式”，而是组合式替代：

1. `AtlantaFX` 负责全局主题和标准控件样式统一
2. `MaterialFX` 负责更接近 Material Design 的交互型控件
3. `ControlsFX` 或自定义实现负责替代 `JFXDrawer`
4. `SVGPath` / `Ikonli` / 自定义图标组件负责替代 `SVGGlyph`
5. 自定义 undecorated stage header 负责替代 `JFXDecorator`

原因：

- `MaterialFX` 更接近 JFoenix 的 Material 组件路线
- `AtlantaFX` 更适合把标准 JavaFX 控件做得现代、统一、可持续
- 目前没有证据表明 `MaterialFX` 或 `AtlantaFX` 任意一方可以单独完整平替 `JFXDrawer`
- `JFXDrawer` 应当被视为一个独立设计点，而不是继续依附某个特定 UI 库

### 8.6 当前建议

- 保持 Material Design 风格方向
- 优先调研并验证 `AtlantaFX + MaterialFX + 自定义 Drawer/Decorator` 的组合方案
- 只有在验证结果明显达不到当前 UI 观感目标时，才回头评估 `us.ihmc:jfoenix`

### 8.7 第三方控件替代原则

- 对关键第三方控件，不能只做“是否有同名替代库”的表面比较
- 当某个控件没有直接替代品时，必须先查看其源码或实际实现方式
- 若发现原控件只使用了部分能力，则优先自定义实现该能力子集
- 若发现现有代码和标准 JavaFX 能覆盖其核心行为，则可以用现有代码重构替换
- 只有在源码分析后仍确认替代成本过高时，才考虑继续保留第三方依赖
- `JFXDrawer`、`JFXDecorator` 这类关键交互组件必须遵循该原则

### 8.8 README 视觉基准总结

基于 `README.md` 与其中展示图片，旧版项目的视觉基准可归纳为：

- 主界面：
  - 顶部蓝色标题栏
  - 左侧固定导航栏与用户信息卡
  - 中部卡片式推荐歌单网格
  - 底部播放控制条始终固定
- 歌单列表页：
  - 左上角专辑封面
  - 右上角歌单标题、标签、简介
  - 下方表格区与搜索框
  - 支持右键菜单
- 详情抽屉页：
  - 从底部展开
  - 左侧黑胶唱片与唱针动画
  - 右侧歌曲信息与歌词滚动
  - 整体是沉浸式大图背景
  - 由底部播放条左侧当前歌曲信息区触发展开
  - 展开后是“大号详情页”而不是小弹层
  - 顶部和底部仍保留主应用视觉连续性
  - 歌词区域需要半透明遮罩层以保证可读性
  - 必须保留明显的“从底部抽出”的动效感知
- 精简模式：
  - 小尺寸悬浮播放器
  - 保留封面、控制按钮、歌词入口
  - 具备桌面控件属性
- 弹窗：
  - 关于弹窗与确认弹窗风格统一
  - 不能退化成系统默认样式

重构要求：

- 新版必须复现这些核心场景
- 抽屉式详情页是高优先级视觉特征，必须重点保留或升级
- 详情抽屉页必须接近你补充图片中的交互效果和视觉层次

## 11. 分阶段实施方案

### Phase 0：确认与冻结

- 明确最终公开版本仍以“用户自有或授权内容 + Spring Boot 4 后端”作为目标
- 旧网易云能力不再作为最终发布方案
- 第一阶段允许以 legacy 兼容层方式迁移旧在线流程，仅用于旧版本功能迁移与新 UI 联调验证
- legacy 在线流程必须被隔离，后续可以整体拆除
- 确定模块拆分方案

### Phase 1：旧版本能力迁移到新 UI

- 改为 Maven 多模块
- 建立 JavaFX 客户端模块骨架
- 抽离公共模型和配置模块
- 统一到 JDK 17
- 旧 `src` 目录下的历史代码在 Phase 1 期间作为迁移输入与行为参考
- 优先把旧版主要交互和页面状态迁入新 UI，先让页面、抽屉、歌词、播放条、歌单与旧功能流程跑通
- 本地能力与旧在线能力在 Phase 1 允许通过兼容适配层接入新 UI，但不得继续向新核心结构扩散
- Phase 1 不允许回退到普通 JavaFX 默认风格
- Phase 1 的目标是“新 UI 能完整跑通旧版本主要功能路径”，不是只做演示级骨架

### Phase 1.1：首批编码目标

- 完成聚合父工程和多模块目录结构
- 建立 `player-common`、`player-domain`、`player-infra`、`player-server`、`player-fx`
- 提供基础配置模型、统一响应模型、应用路径解析工具
- 建立 JavaFX 启动入口和主窗口骨架
- 完成第一版自定义 Drawer 组件，优先覆盖旧项目 `JFXDrawer` 实际使用到的能力子集
- 将旧版底部播放条、歌曲详情抽屉、歌词区域和主要列表页面迁入新 UI
- 通过兼容层接通旧本地能力和旧在线能力，让旧版交互在新 UI 里先跑起来
- Spring Boot 4 后端骨架可以保留，但当前不作为第一阶段的主要集成目标

### Phase 1.2：自定义 Drawer 实现边界

- 自定义 Drawer 仅需覆盖当前项目真实使用场景
- 组件设计应保留向 `LEFT` / `RIGHT` / `TOP` / `BOTTOM` 四方向扩展的能力
- 当前播放器主界面优先使用 `BOTTOM` 方向
- 第一版必须支持遮罩层、平滑展开/收起、点击遮罩关闭、状态回调
- 第一版必须支持大号歌曲详情页容器，不允许退化成普通 `Popup`
- 第一版暂不实现拖拽调整尺寸、边缘悬停激活、四方向切换、迷你抽屉模式
- 若后续确有需要，再按真实场景增量扩展

### Phase 1.3：歌词迁移要求

- 新 UI 的歌词区必须保留旧版“歌词滚动”的动态感，不能只做静态高亮切换
- 歌词区域必须有真实视口裁剪，超出区域的文本不允许穿出容器
- 当前行、相邻行、远处行的视觉层次必须明确区分
- 抽屉歌词滚动应优先参考旧版 `MainApp.java` 的过渡思路，但要避免粗糙位移和潜在泄漏问题
- 后续接入真实播放时间时，歌词滚动逻辑必须可以从演示驱动平滑切换到播放进度驱动

## 12. 当前实现快照

### 12.1 已落地的工程骨架

### 12.4 本轮实现原则

- 当前按“两阶段迁移”执行：先让新 UI 跑通旧版本主要功能流程，再让 Spring Boot 4 后端接管
- 先保证结构、边界、动效方向和视觉层次成立
- 第一阶段允许使用演示数据驱动部分 UI，但必须逐步被旧功能真实流程替换
- 旧在线能力在第一阶段只能作为 legacy 兼容层存在，不代表最终技术路线
- 当前优先让主界面布局、歌单页结构、左侧入口和抽屉歌词交互先接近旧版
- 后续若要继续替换 `JFoenix` 其余控件，仍必须遵循“先看源码，再决定复用或自研”的原则

### Phase 2：Spring Boot 4 后端接管

- 将 Phase 1 中已跑通的旧本地能力迁入 Spring Boot 4
- 将歌词读取、封面读取、歌单与播放状态改为由后端服务承载
- 将 SQLite / MySQL / PostgreSQL 支持真正收敛到后端
- 将 Phase 1 中的 legacy 旧在线兼容层拆除或替换

### Phase 3：重做桌面端

- 拆分视图层、控制层、服务层
- 建立清晰的 UI 状态模型
- 优化动画、托盘、迷你模式、桌面歌词
- 建立跨平台文件操作封装

### Phase 4：质量与发布

- 增加单元测试和集成测试
- 增加 Sonar 规则修复
- 增加数据库切换配置
- 验证 Windows 与 macOS 运行

## 13. 我的执行原则

后续如果由我继续维护，我会遵循以下原则：

- 每次开始改动前先看本需求文档
- 不提交代码，除非你明确要求
- 优先做架构清理，不做表面升级
- 先稳定 JDK 17，再评估 JDK 21
- 优先去除历史包袱，尤其是高风险在线能力和高耦合 UI 代码
- 优先保持 Material Design 风格连续性，除非替代方案明确更优
- 遇到关键第三方控件时，先读源码，再决定复刻子集、重写还是保留依赖
- 每轮不仅要编译通过，还要做可启动验证
- 完成启动验证后必须主动关闭应用进程，不能把验证程序留给你手动收尾
